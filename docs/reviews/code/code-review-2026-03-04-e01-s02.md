## CRITICAL Issues (Confidence 90-100)

### 1. Dual Cookie System Creates Session Desynchronization (Confidence: 95)

**File:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/src/app/actions/auth.ts`, lines 34-46

**Problem:** The `loginAction` manually sets cookies named `sb-access-token` and `sb-refresh-token` **on top of** the cookies that `@supabase/ssr` already manages automatically. The `@supabase/ssr` library (v0.8.x) manages its own cookie storage internally via the `getAll`/`setAll` callbacks configured in `server.ts` and `middleware.ts`. These Supabase-managed cookies use a different naming convention (typically `sb-<project-ref>-auth-token` chunked across multiple cookies).

The result is two parallel cookie systems:
- The manual `sb-access-token` / `sb-refresh-token` cookies set in `auth.ts`
- The automatic Supabase-managed cookies set by `@supabase/ssr` via the `setAll` callback

The middleware and auth context call `supabase.auth.getSession()` which reads from the **Supabase-managed** cookies, not from the manually-set ones. This means:
- The manual cookies are never read by anything
- The "Remember me" `maxAge` configuration (lines 31, 38, 45) has **no effect** -- Supabase controls its own cookie expiry via its own session configuration
- On logout, `logoutAction` deletes the manual cookies (line 56-57), but the real session cookies managed by `@supabase/ssr` are what actually matter for the `signOut()` call

**Fix:** Remove the manual cookie management entirely (lines 31-46 in `loginAction`, lines 55-57 in `logoutAction`). The `@supabase/ssr` library handles cookies through the `setAll` callback. To implement "Remember me" with different session durations, configure it at the Supabase project level or pass session configuration to `signInWithPassword`.

```typescript
// auth.ts - simplified
export async function loginAction(email: string, password: string, rememberMe: boolean = false) {
  const supabase = await createClient()
  const { data, error } = await supabase.auth.signInWithPassword({ email, password })

  if (error) {
    return {
      success: false,
      error: { code: error.code, message: 'Invalid email or password' },
    }
  }

  return { success: true }
}
```

### 2. Middleware Uses `getSession()` Instead of `getUser()` for Auth Verification (Confidence: 92)

**File:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/middleware.ts`, line 29

```typescript
const { data: { session } } = await supabase.auth.getSession()
```

**Problem:** The Supabase documentation explicitly warns that `getSession()` reads the session from the cookie/storage **without validating the JWT with the Supabase Auth server**. This means a client could craft or modify the JWT in the cookie and `getSession()` would still return it as valid. For server-side route protection, `getUser()` should be used instead, as it sends the access token to the Supabase Auth server for validation.

From the Supabase docs: "Never trust `supabase.auth.getSession()` inside server code...It isn't guaranteed to revalidate the Auth token. It reads all the items from the storage medium, including the auth token, and if found, it attempts to use the refresh token to get a new access token."

**Fix:** Replace `getSession()` with `getUser()` in the middleware:

```typescript
const { data: { user } } = await supabase.auth.getUser()

if (!user && request.nextUrl.pathname.startsWith('/dashboard')) {
  return NextResponse.redirect(new URL('/login', request.url))
}

if (user && request.nextUrl.pathname === '/login') {
  return NextResponse.redirect(new URL('/dashboard', request.url))
}
```

### 3. Server Action Lacks Server-Side Input Validation (Confidence: 93)

**File:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/src/app/actions/auth.ts`, lines 7-10

```typescript
export async function loginAction(
  email: string,
  password: string,
  rememberMe: boolean = false
) {
```

**Problem:** The server action accepts `email` and `password` as raw string parameters without any server-side validation. The Zod validation in `LoginForm.tsx` only runs on the client. Server actions are publicly accessible HTTP endpoints -- an attacker can call them directly, bypassing the client-side form entirely. The implementation plan (section on "Security Considerations") explicitly states: "Validate all inputs server-side (don't trust client)."

Without server-side validation, malformed or excessively long strings could be passed directly to `supabase.auth.signInWithPassword()`.

**Fix:** Validate inputs with the same Zod schema server-side:

```typescript
import { loginSchema } from '@/lib/schemas/login-schema'

export async function loginAction(email: string, password: string, rememberMe: boolean = false) {
  const parsed = loginSchema.safeParse({ email, password, rememberMe })
  if (!parsed.success) {
    return { success: false, error: { code: 'VALIDATION_ERROR', message: 'Invalid input' } }
  }

  const supabase = await createClient()
  const { data, error } = await supabase.auth.signInWithPassword({
    email: parsed.data.email,
    password: parsed.data.password,
  })
  // ...
}
```

### 4. Middleware Only Protects `/dashboard`, Leaves `/trips/*` Unprotected (Confidence: 90)

**File:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/middleware.ts`, line 32

```typescript
if (!session && request.nextUrl.pathname.startsWith('/dashboard')) {
```

**Problem:** The middleware only protects routes starting with `/dashboard`. However, the application has routes under `/trips/[tripId]/*` (including itinerary, budget, voting pages) that contain user trip data and should also be protected. An unauthenticated user can directly navigate to `/trips/some-id/itinerary` and the middleware will not redirect them to login.

**Fix:** Extend the protection to cover all authenticated routes:

```typescript
const protectedPrefixes = ['/dashboard', '/trips', '/settings', '/profile']
const isProtectedRoute = protectedPrefixes.some(prefix =>
  request.nextUrl.pathname.startsWith(prefix)
)

if (!user && isProtectedRoute) {
  return NextResponse.redirect(new URL('/login', request.url))
}
```

## IMPORTANT Issues (Confidence 80-89)

### 5. Auth Context Creates New Supabase Client on Every Render (Confidence: 85)

**File:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/src/lib/auth-context.tsx`, line 18

```typescript
export function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<User | null>(null)
  const [loading, setLoading] = useState(true)
  const supabase = createClient()  // <-- called every render
```

**Problem:** `createClient()` is called directly in the component body, meaning it runs on every render of `AuthProvider`. While `@supabase/ssr`'s `createBrowserClient` may internally deduplicate clients, this is an implementation detail that should not be relied upon. It also means the `supabase` reference changes on every render, which triggers the `useEffect` dependency `[supabase]` on line 36, potentially causing the `onAuthStateChange` subscription to be torn down and re-established on every render.

**Fix:** Memoize or use `useState` to create the client once:

```typescript
const [supabase] = useState(() => createClient())
```

Or remove `supabase` from the dependency array if `createBrowserClient` is known to return a singleton.

### 6. "Remember Me" Feature Does Not Actually Work (Confidence: 88)

**Files:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/src/app/actions/auth.ts`, line 31; Story document, line 30

**Problem:** This is a direct consequence of Issue #1. The story's acceptance criterion states: *"Remember me" checkbox extends session to 30 days*. The implementation sets `maxAge` on manually-created cookies that are never read by anything. The actual session duration is controlled by the Supabase project configuration (JWT expiry + refresh token rotation), not by these orphaned cookies.

The feature appears to work in the UI (checkbox exists, value is passed through) but has zero effect on actual session duration.

**Fix:** Implementing variable session duration requires either:
1. Configuring Supabase's JWT expiry at the project level (affects all sessions uniformly), or
2. Using a custom approach where the server tracks "remember me" preference and adjusts refresh token handling accordingly, or
3. Documenting this as a known limitation if Supabase does not support per-session expiry configuration.

**Files:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/src/components/auth/LoginForm.tsx`, line 46; `/Volumes/SSD/Dev/Asia Trip/tripflow-next/src/app/actions/auth.ts` (return value)

**Problem:** There is no `src/app/dashboard/` directory or `page.tsx` in the codebase. The existing app structure has a home page at `/` and trip detail pages at `/trips/[tripId]/*`, but no dashboard route.

**Fix:** Either create a minimal `/dashboard` page, or redirect to `/` (the existing home page) after successful login:

```typescript
router.push('/')
```

### 8. E2E Test Hardcodes Base URL (Confidence: 80)

**File:** `/Volumes/SSD/Dev/Asia Trip/tripflow-next/e2e/test-homepage-login.spec.ts`, lines 6, 66

```typescript
await page.goto('http://localhost:3001/', { waitUntil: 'networkidle' });
// ...

```

**Problem:** This test file hardcodes `http://localhost:3001` instead of using Playwright's `baseURL` configuration. If the dev server runs on a different port or the Playwright config specifies a different base URL, these tests will fail or test the wrong instance. The other E2E test (`story-e01-s02.spec.ts`) correctly uses relative paths like `page.goto('/login')`.

**Fix:** Use relative URLs and rely on Playwright's `baseURL`:

```typescript
await page.goto('/', { waitUntil: 'networkidle' });
// ...
await page.goto('/login');
```

## Summary

The authentication implementation follows a generally sound architecture (server actions, middleware, cookie-based sessions, Zod validation, accessible form markup). However, there are **4 critical issues** that need to be addressed before this story can be considered complete:

1. **Dual cookie system** -- The manual cookie management conflicts with `@supabase/ssr`'s built-in cookie handling, rendering the "Remember me" feature non-functional and creating confusion about which cookies are authoritative.
2. **`getSession()` in middleware** -- Must use `getUser()` for server-side JWT validation to prevent token forgery.
3. **No server-side input validation** -- Server actions are public endpoints and must validate inputs independently of the client.
4. **Incomplete route protection** -- The `/trips/*` routes are unprotected by the middleware.

Fixing these four issues is essential for the security and correctness of the authentication flow. Issues #5-#8 are important quality/correctness concerns that should also be addressed.
